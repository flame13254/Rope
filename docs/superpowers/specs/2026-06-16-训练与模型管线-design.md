# 训练与模型管线设计（Phase 1）

> 智能排绳故障检测与数字孪生系统 —— 模型训练侧代码框架
> 日期：2026-06-16　状态：已确认设计，待评审 spec

## 1. 背景与范围

当前**尚无真实视频/图像数据**，本阶段只交付一套**可端到端运行的训练侧代码框架**；真实数据到位后替换数据集即可正式训练。

**本 spec 只覆盖 Phase 1（训练与模型管线）。** 边缘推理进程、FFmpeg 转发、MQTT、上位机（LabVIEW）属于 Phase 2，另起 spec。

**运行位置说明**：训练运行在带 GPU 的开发/训练机上；"代码要能跑在大部分工控机上"是 Phase 2 边缘**推理**侧的约束，本阶段通过"导出 onnx/openvino/engine 多后端 + 后端可切换"为其铺路。

## 2. 已确认的关键决策

| 决策点 | 结论 |
|---|---|
| 检测范式 | **分类为主 + OBB 备选**：主推理用图像分类（正常/故障→0/1）；保留 OBB 旋转框训练做对比/未来定位扩展 |
| 推理后端 | **后端可切换**：NVIDIA→TensorRT、Intel→OpenVINO、纯 CPU→ONNXRuntime |
| 类别 | **单类别** `{0: fault}`，二分类语义（有故障 / 无故障） |
| 标注策略 | **只标 OBB**（8 点旋转框）；脚本自动派生分类标签（有框=故障，无框=正常），一次标注喂两条管线 |
| 模型 | YOLOv8s 与 YOLO11s，各训"分类 + OBB"共 4 个；**删除 `yolo26n.pt`** |
| 代码结构 | **A. 配置驱动的实验运行器**：`experiments.yaml` 列出"模型×任务"，脚本统一遍历 |

## 3. 目录与文件

```
configs/experiments.yaml      # 4 个 run：v8s-cls / v11s-cls / v8s-obb / v11s-obb（共享超参 + 可逐项覆盖）
rope_obb.yaml                 # OBB 数据集配置，单类别 {0: fault}
tools/
  make_dummy_dataset.py       # 生成少量合成图（平行纹理=正常 / 交叉=故障）+ OBB 标注，验证管线
  derive_cls_dataset.py       # 读 OBB 标注 → 派生分类目录树 dataset_cls/{train,val}/{normal,fault}/
  copy_paste_aug.py           # 离线增强：抠真实故障块随机贴到正常背景，造故障样本 + 重生成 OBB 标注
train.py                      # 配置驱动：依次训练 4 个模型
export.py                     # best.pt → onnx + openvino +（检测到 CUDA 才）engine/FP16
benchmark.py                  # 各模型×后端 实测 F1 / 单帧 ms / 显存 MB → 输出对比表（md + csv）
```

## 4. 各组件职责（单一职责、可独立运行）

### 4.1 `rope_obb.yaml`
```yaml
path: ./dataset
train: images/train
val: images/val
names: {0: fault}
```
OBB 任务由 `-obb` 模型隐含；标注为单类别 8 点旋转框。

### 4.2 `configs/experiments.yaml`
- `common`：epochs、imgsz、batch、device、project 等共享超参。
- `models`：列表，每项 `{name, weights, task(classify|obb), data, 可选 imgsz/epochs 覆盖}`。
- 默认 4 项：`v8s-cls(yolov8s-cls.pt)`、`v11s-cls(yolo11s-cls.pt)`、`v8s-obb(yolov8s-obb.pt)`、`v11s-obb(yolo11s-obb.pt)`。
- **device 自动**：配置可写 `auto`，运行时有 CUDA 用 GPU、否则回退 CPU。

### 4.3 `tools/make_dummy_dataset.py`
- 用 OpenCV/PIL 程序化生成数十张合成图：正常=平行斜纹，故障=叠加交叉线/隆起斑块。
- 同步生成 OBB 标注（故障图含框，正常图空标注）。
- 输出到 `dataset/`，让无真实数据时也能跑通全链路。**仅用于验证，不进入正式训练。**

### 4.4 `tools/derive_cls_dataset.py`
- 遍历 `dataset/labels/{train,val}`：标注非空→该图归 `fault`，空/无标注→`normal`。
- 在 `dataset_cls/{train,val}/{normal,fault}/` 下建立图片（软链或复制），供 YOLO 分类（按文件夹分类）使用。
- 幂等：可重复运行覆盖。

### 4.5 `tools/copy_paste_aug.py`（对应方案第八节）
- 从 OBB 标注的故障图中按框抠出故障小块；随机位置（可小角度、轻微缩放）贴到正常背景帧。
- 重新生成对应 OBB 标注。**禁用大角度旋转**（保平行纹理朝向恒定，避免模型学偏）。
- 离线生成增广数据写入训练集。

### 4.6 `train.py`
- 读 `experiments.yaml`，对每个 model 调用 `YOLO(weights).train(...)`。
- 分类 `data=dataset_cls`，OBB `data=rope_obb.yaml`。
- 在线增强：HSV/亮度/对比度/模糊/噪声开启以抗环境光；`degrees=0`（禁大角度旋转），保留 `translate`、轻微 `scale`、`fliplr`。
- 分类 imgsz 用 320/640（保螺旋纹理，不用默认 224），OBB 640+；可按模型覆盖。
- 结果存 `runs/<name>/`。

### 4.7 `export.py`
- 对每个 run 的 `best.pt` 导出：`onnx`（通用 CPU）、`openvino`（Intel）、`engine`（TensorRT/FP16，仅在检测到 CUDA 时）。
- 缺少某后端依赖时跳过并告警，不中断其余导出。

### 4.8 `benchmark.py`（对应方案第七节"模型冗余设计"）
- 对每个 模型×可用后端：在 val 集算 F1（准确度），测单帧推理耗时（ms，多次取均值），测显存/内存占用（MB）。
- 汇总成对比表输出 `runs/benchmark.md` + `runs/benchmark.csv`，作为"选哪个模型打包进主进程"的依据。

## 5. 数据流

```
make_dummy_dataset.py ──→ dataset/(images+OBB labels, 单类fault)
                              │
          derive_cls_dataset.py ──→ dataset_cls/{train,val}/{normal,fault}/
          copy_paste_aug.py    ──→ 扩充 dataset/ 故障样本
                              │
   experiments.yaml ──→ train.py ──→ runs/<name>/weights/best.pt  (×4)
                              │
                    export.py ──→ best.onnx / best_openvino/ / best.engine  (×可用后端)
                              │
                   benchmark.py ──→ runs/benchmark.{md,csv}（F1 / ms / MB 对比）
```

## 6. 错误处理
- 缺依赖（openvino/tensorrt）→ 跳过该后端 + 明确告警，继续其余。
- 无 CUDA → device 回退 CPU；TensorRT 导出自动跳过。
- 数据集为空/路径不存在 → 明确报错并提示先跑 `make_dummy_dataset.py` 或放入真实数据。
- `experiments.yaml` 缺字段 → 校验并给出可读错误。

## 7. 验证（可运行性）
端到端冒烟：`make_dummy_dataset.py` → `train.py`（1–2 epoch）→ `export.py` → `benchmark.py` 全部跑完不报错，即证明框架可用。真实数据到位后替换 `dataset/` 内容即可正式训练，无需改代码。
